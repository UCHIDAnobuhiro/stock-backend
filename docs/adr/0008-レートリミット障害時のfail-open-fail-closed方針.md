# ADR-0008: レートリミット障害時の fail-open / fail-closed 方針

| 項目       | 内容       |
| ---------- | ---------- |
| ステータス | Accepted   |
| 日付       | 2026-07-20 |

---

## コンテキスト

`internal/transport/httpratelimit` の `Limiter.Allow()` は、Redis が未接続（`rdb == nil`）
または Lua スクリプト実行がエラーになった場合、一律で `slog.Warn` のログを出したうえで
`Result{Allowed: true}` を返し、リクエストを許可していた（fail-open）。これは
`candles.CachingRepository` のキャッシュや JWT blacklist（ログアウト時の即時失効）といった
非クリティカルな Redis 利用と同じ「グレースフルデグレード」方針を、レートリミット全体に
一律適用したものだった。

しかし signup / login（IP・メール両方）/ OAuth コールバックのレートリミットは、
ブルートフォース攻撃・アカウント列挙・OAuth コールバック濫用を防ぐためのセキュリティ制御である。
Redis 障害時に fail-open すると、この防御が無効化されたままアプリケーションは正常に
動作し続けてしまい、攻撃者は Redis 障害（あるいは意図的な Redis への攻撃）に乗じて
レートリミットなしでログイン試行を行えてしまう（issue #266）。

一方、OAuth 機能は `internal/app/di/oauth.go` の `NewOAuthHandler` が Redis 必須とし、
`rdb == nil` の場合はそもそも OAuth ハンドラーを起動しない（= fail-closed 相当）実装に
なっており、同じ Redis 障害に対して「レートリミットは fail-open」「OAuth state 保存は
fail-closed」という矛盾した方針が同居していた。

## 決定

セキュリティクリティカルなレートリミット（signup / login IP / login email /
OAuth コールバック IP）は **fail-closed** とする。Redis 未接続・障害時はリクエストを
拒否し、HTTP 503（`ErrorResponse{Error: "service temporarily unavailable"}`）を返す。

このため `httpratelimit.Limiter.Allow()` に `Policy`（`FailOpen` / `FailClosed`）を追加し、
呼び出し側が用途ごとに選択できるようにした。`Result` には `ServiceUnavailable bool` を
追加し、呼び出し側が 429（レートリミット超過）と 503（基盤障害による拒否）を区別できる
ようにした。503 応答では `Retry-After` ヘッダーを付与しない（障害復旧時間が不明なため）。

非クリティカルな Redis 利用（`candles.CachingRepository` のキャッシュ、JWT blacklist
によるログアウト時の即時失効）は、可用性を優先し **fail-open**（グレースフルデグレード）
を維持する。

`Policy` のゼロ値は **FailClosed** とする（secure by default）。呼び出し側が `Policy` の
指定を忘れた場合でも安全側（拒否）に倒れるようにするための設計であり、`FailOpen` は
非クリティカルな用途に対する明示的なオプトインとして扱う。

### 使い分け基準

| 用途                                          | 方針         | 理由                                                       |
| --------------------------------------------- | ------------ | ------------------------------------------------------------ |
| signup / login（IP・メール） / OAuth コールバック | FailClosed   | ブルートフォース・アカウント列挙対策として必須の防御         |
| candles キャッシュ                            | FailOpen     | キャッシュ欠落は性能劣化のみで、正確性・安全性に影響しない   |
| JWT blacklist（ログアウト）                   | FailOpen     | Cookie 削除自体は継続でき、JWT は元々短命（1時間）で失効する |

## 理由

| 観点             | FailClosed の優位性                                                       |
| ---------------- | -------------------------------------------------------------------------- |
| セキュリティ      | Redis 障害を突いたブルートフォース攻撃・アカウント列挙を防止できる         |
| 一貫性           | OAuth（state 保存必須・fail-closed 相当）と方針が揃い、矛盾がなくなる      |
| 可観測性         | 503 は運用者に Redis 障害を明示的に伝え、429（クライアント起因）と区別できる |
| 影響範囲の限定   | Policy を用途ごとに指定するため、キャッシュ等の非クリティカル用途は従来どおり可用性優先を維持できる |

## 代替案

| 代替案                                             | 不採用の理由                                                                 |
| --------------------------------------------------- | ----------------------------------------------------------------------------- |
| 全レートリミットを一律 fail-closed にする           | candles 等の非クリティカルな用途まで Redis 障害時に機能停止させてしまう       |
| in-memory フォールバック（ローカルレートリミット）  | 複数インスタンス間でカウンタを共有できず、水平スケール時に実効的な制限にならない。実装・検証コストも大きい |
| Redis を Sentinel/Cluster 化して SPOF 自体を解消する | インフラ変更が必要で本 ADR のスコープを超える。将来的な改善として別途検討する |

## 影響

### ポジティブな影響

- Redis 障害時でもブルートフォース・アカウント列挙対策が有効なまま維持される。
- OAuth（fail-closed 相当）とレートリミットの方針が揃い、セキュリティ設計の一貫性が向上した。
- 429 と 503 が区別されるため、監視・アラート設計がしやすくなった。

### ネガティブな影響・トレードオフ

- Redis が signup / login / OAuth コールバックの可用性における SPOF（単一障害点）になる。
  Redis 障害時はこれらのエンドポイントが 503 を返し、新規ユーザー登録・ログイン・
  OAuth ログインができなくなる。
- 運用上、Redis の死活監視・アラート設定の重要性が増す（Redis 障害 = 認証系エンドポイント
  停止に直結するため）。
- 既存クライアントは 503 応答（Retry-After なし）へのハンドリングが必要になる
  （429 と異なり再試行タイミングの目安を提供しない）。
