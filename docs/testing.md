# テスト方針

このリポジトリでテストを書く・レビューするときの指針をまとめる。
「どう書くか」の機械的なルール（テーブル駆動、モック定義の形式等）は
`.claude/skills/test-generate/SKILL.md` にあるため、ここでは
**何をテストすべきか / 何をテストすべきでないか** という判断基準を中心に書く。

## 1. テストの目的

テストは「コードを書いた証拠」ではなく、**将来の変更を安全にするための資産**。
良いテストは次の2つを同時に満たす：

- **コードを壊したら落ちる**（検出力がある）
- **コードを壊していないリファクタリングでは落ちない**（実装に密結合していない）

この2つ目が特に重要。実装の内部構造に密結合したテストは、正しいリファクタリング
のたびに書き直しが必要になり、「テストを直すコスト > テストが守ってくれる価値」
になって負債化する。

**原則: 実装ではなく「振る舞い（入力→出力・観測可能な副作用）」をテストする。**

## 2. レイヤー別戦略（現状の構成）

このリポジトリは既にレイヤーごとにテスト手法を使い分けており、この方針を維持する。

| レイヤー | 手法 | 実例 |
|---------|------|------|
| ドメインロジック（純粋関数） | 入出力のテーブル駆動テスト。モック不要 | `internal/feature/candles/aggregation_test.go` |
| usecase | リポジトリを関数型フィールドのモックに差し替え | `internal/feature/candles/usecase_test.go` |
| repository | testcontainers で**実 PostgreSQL** を起動し、テストごとに隔離 DB を作成 | `internal/feature/watchlist/repository_test.go` |
| HTTP handler | `httptest` + chi + usecase モック。ステータスコードとレスポンス JSON を検証 | `internal/feature/watchlist/watchlisthttp/handler_test.go` |
| 外部 API アダプタ | `httptest.Server` で外部 API を偽装 | `internal/feature/candles/twelvedata/repository_test.go` |
| Redis 依存 | `miniredis` / `redismock` | `internal/feature/candles/caching_repository_test.go` |

ポイント：

- **repository テストで DB をモックしない。** SQL の正しさ（制約違反、ORDER BY、
  FK エラーの変換等）は実 DB でしか検証できない。`sql.DB` をモックしたテストは
  「自分が書いた SQL 文字列がそのまま渡ること」しか確認できず、価値がほぼない。
- **usecase テストで DB を使わない。** usecase の関心は「リポジトリをどう組み合わせ、
  エラーをどう扱うか」であり、モックで十分かつ高速。
- **handler テストは HTTP の言葉で検証する。** ステータスコード・レスポンスボディ・
  ヘッダーが契約。内部でどのメソッドが呼ばれたかは原則気にしない。

## 3. 書くべきテスト

優先度の高い順：

1. **分岐のあるビジネスロジック** — 集計、バリデーション、エラー変換など。
   例: `AggregateWeekly` の「日曜は前の月曜に丸める」（`aggregation_test.go`）。
   仕様がコメントとテストケース名で読み取れるのが理想。
2. **エラーパス** — リポジトリがエラーを返したとき usecase / handler が何を返すか。
   センチネルエラー（`ErrAlreadyInWatchlist` 等）は `errors.Is` で検証する。
   実例: `watchlist/repository_test.go` の `Add_DuplicateEntry` / `Add_UnknownSymbol`。
3. **境界値** — 空スライス、0件、存在しない ID、他ユーザーのデータが見えないこと
   （例: `TestWatchlistRepository_ListByUser_Isolation`）。
4. **セキュリティに関わる振る舞い** — 認証ミドルウェア、CSRF、レートリミッタ。
   落ちたときの被害が大きいので厚めに書いてよい。
5. **バグ修正時の再発防止テスト** — バグを再現するテストを先に書き、修正で緑にする。
   このテストは「なぜこの仕様なのか」のドキュメントとして最も価値が高い。

## 4. 書かないほうがいいテスト

AI にテストを生成させると、以下のパターンが混入しやすい。レビューで弾くこと。

### 4.1 トートロジー（同語反復）テスト

モックに設定した値が、モックからそのまま返ることを確認しているだけのテスト。

```go
// ダメな例: これは usecase ではなくモックをテストしている
mockRepo.FindFunc = func(...) ([]Candle, error) { return fixture, nil }
got, _ := uc.GetCandles(...)
assert.Equal(t, fixture, got) // usecase が何もしないなら常に通る
```

usecase が入力をそのまま横流しするだけの薄い層なら、テストの価値は
「リポジトリに渡るパラメータが正しいこと」と「エラーがそのまま伝播すること」
の2点だけに絞る（現状の `candles/usecase_test.go` はこの絞り方ができている）。
横流し確認のためだけにケースを何個も増やさない。

### 4.2 自動生成コードのテスト

`internal/api/types.gen.go`（oapi-codegen）や `internal/feature/*/sqlc/`（sqlc）は
生成物なのでテスト不要。生成コードの正しさはジェネレータの責任で、
その利用側（repository / handler）のテストで間接的にカバーされる。

### 4.3 実装詳細に密結合したテスト

- **呼び出し回数の過剰検証**: 「1回だけ呼ばれること」が仕様として意味を持つ場合
  （例: キャッシュヒット時に DB へ行かないこと）だけ検証する。
  意味なく全メソッドの `XxxCalls` をアサートすると、内部の処理順を変えただけで落ちる。
- **プライベート関数の直接テスト**: エクスポートされた関数経由で分岐を通せるなら
  そちらを使う。どうしても通せない複雑なロジックは、まず関数の切り出し方を疑う。
- **ログ出力の文字列一致**: ログはデバッグ用であり契約ではない。
  マスク処理（`infra/logging`）のような「ログ自体が仕様」の場合のみ例外。

### 4.4 標準ライブラリ・フレームワークのテスト

「chi がルーティングすること」「encoding/json がエンコードすること」自体の確認は不要。
handler テストはあくまで**自分が書いた分岐**（バリデーション、エラー→ステータス変換）
を通すために書く。

### 4.5 網羅風だが分岐を増やさないケース

テーブル駆動テストでケースを増やすのは簡単だが、**同じコードパスを通るケースは
何個あっても検出力は増えない**。「AAPL で成功」「GOOG で成功」「MSFT で成功」の
3ケースは1ケースと同じ価値。ケースを足すときは「このケースはどの分岐・境界を
新たに通すのか」を自問する。

## 5. AI 生成テストのレビューチェックリスト

マージ前に最低限これだけ確認する：

- [ ] **このテストは、対象コードにバグを仕込んだら落ちるか？**
      疑わしければ実際にプロダクションコードを1行壊して `go test` してみる
      （簡易ミューテーションテスト）。落ちなければそのテストは飾り。
- [ ] アサーションがモックの設定値のコピーになっていないか（§4.1）
- [ ] エラー検証が `assert.Error(t, err)` だけで済まされていないか。
      **どのエラーか**まで `assert.ErrorIs` / `errors.Is` で見る
- [ ] ケース名（英語）が「何を検証するか」を説明しているか。
      `"success: case 1"` のような名前は仕様を語っていない
- [ ] `t.Parallel()` と `t.Setenv` が同一テストで併用されていないか（panic する）
- [ ] 同じコードパスの重複ケースで水増しされていないか（§4.5）
- [ ] repository テストが他のテストとデータを共有していないか
      （`dbtest.OpenIsolatedDB` でテストごとに隔離するのが本リポジトリの流儀）

## 6. カバレッジについて

カバレッジは**低いことの警告**には使えるが、**高いことの証明**には使えない
（アサーションのないテストでも数字は上がる）。数値目標は置かず、
「§3 に挙げた優先度の高い箇所が漏れていないか」の点検に使う。

```bash
go test ./... -race -cover
```
